## Шаг 1 — Создать Firebase проект

1. Перейдите на [console.firebase.google.com](https://console.firebase.google.com)
2. **Add project** → введите название (напр. `cyprusguard`)
3. Google Analytics → можно отключить → **Create project**

---

## Шаг 2 — Включить Realtime Database

1. В левом меню: **Build → Realtime Database**
2. **Create database** → выберите регион **Europe (belgium)** → **Start in test mode**
3. Скопируйте URL базы данных: `https://cyprusguard-default-rtdb.europe-west1.firebasedatabase.app`

**Правила безопасности** (вставьте в Rules):
```json
{
  "rules": {
    ".read": "auth != null",
    ".write": "auth != null",
    "clients": {
      ".read": true,
      ".indexOn": ["accessToken", "tgChatId", "tg"]
    }
  }
}
```

---

## Шаг 3 — Включить Authentication

1. **Build → Authentication → Get started**
2. Вкладка **Sign-in method** → включите **Email/Password**
3. Вкладка **Users → Add user**:
   - Email: `admin@cyprusguard.com`
   - Password: придумайте надёжный пароль

---

## Шаг 4 — Включить Storage

1. **Build → Storage → Get started** → **Start in test mode**
2. Регион: `europe-west` → **Done**

---

## Шаг 5 — Вставить конфиг в firebase-config.js

1. **Project Settings (⚙️) → General → Your apps → Add app → Web (</> )**
2. Введите название (напр. `CyprusGuard Web`) → **Register app**
3. Скопируйте объект `firebaseConfig`
4. Откройте `firebase-config.js` и замените блок в начале файла:

```javascript
const firebaseConfig = {
  apiKey:            "AIzaSy...",
  authDomain:        "cyprusguard.firebaseapp.com",
  databaseURL:       "https://cyprusguard-default-rtdb.europe-west1.firebasedatabase.app",
  projectId:         "cyprusguard",
  storageBucket:     "cyprusguard.appspot.com",
  messagingSenderId: "123456789",
  appId:             "1:123...:web:abc..."
};
```

## Шаг 6 — Задеплоить фронтенд

### Вариант A: GitHub Pages (бесплатно, просто)

1. Добавьте все файлы в GitHub репозиторий
2. **Settings → Pages → Source: Deploy from branch → main → / (root)**
3. Через 2-3 минуты сайт будет на: `https://username.github.io/repo-name/`

### Вариант B: Netlify (рекомендуется — поддерживает PWA лучше)

1. Зайдите на [netlify.com](https://netlify.com)
2. **Add new site → Import from Git → GitHub**
3. Выберите репозиторий → **Deploy site**
4. Домен будет: `https://your-site.netlify.app`

### Вариант C: Firebase Hosting (в комплекте с Firebase)

```bash
npm install -g firebase-tools
firebase login
firebase init hosting
# Public directory: . (точка)
# Single-page app: No
firebase deploy
```

## Часто задаваемые вопросы

**Q: Клиент не видит свои данные**
A: Убедитесь, что в разделе «Клиенты» указан правильный clientId у объектов

**Q: Telegram бот не отвечает**
A: Проверьте логи на Railway/Render и правильность переменных окружения

**Q: Фото не загружаются**
A: Убедитесь, что Storage включён и правила разрешают запись

**Q: PWA не устанавливается на iOS**
A: iOS требует HTTPS. Убедитесь, что сайт деплоен (не открывается как file://)

## 🆕 Что нового в этой версии

### Финансы
- **Реальное время** — счета и KPI обновляются мгновенно (раньше только при заходе на вкладку).
- **Статусы счетов** — кнопка «Оплачен / Сбросить» прямо в списке.
- **Авто-просрочка** — неоплаченные счета прошлых месяцев автоматически помечаются «Просрочен».
- **Генерация счетов за месяц** — кнопка «＋ Счета за месяц» создаёт счета по всем объектам (без дублей).
- **PDF-счёт** — кнопка «📄 PDF» открывает печатную форму счёта (печать / сохранение в PDF).
- **График доходности** строится из реально оплаченных счетов; пунктиром показывается прогноз, пока истории нет.

### Объекты и визиты
- **Карточка объекта** — полноценная модалка: статистика, контакты клиента, история визитов и отчётов, кнопки «Изменить» / «Удалить» / «＋ Визит».
- **Карточка визита** — модалка с задачами, заметками, статусом, переходом к отчёту и удалением.
- **Кликабельный календарь** — клик по дню открывает визит или форму создания на эту дату.
- **Календарь** открывается на текущем месяце (раньше был жёстко зашит май 2026).
- Визит закрывается только после сохранения отчёта (убрана двойная логика).

### Фото и хранилище
- Фотографии в отчётах **сжимаются** перед сохранением (общий модуль `compressImage`) — база не раздувается на бесплатном плане Firebase.
- Удаление фото из формы исправлено (раньше при удалении среднего фото удалялось не то).

### Клиентский портал
- **4 языка интерфейса** — RU / EN / DE / FR, переключатель в шапке. Язык берётся из карточки клиента и сохраняется при смене.
- Даты, статусы, состояния и описания тарифов локализованы.

### Telegram-бот
- **Напоминание за 24 часа** до визита (ежечасная проверка, учитывает тумблер «Напоминание за 24ч»).
- Срочные заявки помечаются 🚨 в уведомлении администратору.

### Прочее
- **Живая погода** в Лимассоле на дашборде (Open-Meteo, без API-ключа).
- Service Worker обновлён до стратегии **Network-First** для кода — обновления подхватываются без переустановки PWA (версия кэша `v3`).

## 🛠 Итерация 3 — исправления

- **Календарь**: клик по дню с несколькими визитами теперь открывает список выбора (а не первый визит).
- **Заявки**: исправлен цикл статусов. Новая → В работе → Выполнена; у выполненной кнопка теперь «↺ Вернуть в работу» (раньше ошибочно показывалась «Принять в работу»).
- **Финансы (мобильные)**: строка счёта стала адаптивной — на узких экранах статус и кнопки «Оплачен/Сбросить» и «PDF» переносятся на отдельный ряд и больше не вылезают за край.
- **Telegram-бот**: при старте сбрасывается webhook (частая причина «бот молчит»), добавлены понятные диагностические сообщения в лог (конфликт 409 двух копий, неверный токен 401, сетевые сбои). Если бот не отвечает — смотрите логи хостинга, теперь причина видна сразу.

### Если бот всё ещё молчит — проверьте по порядку:
1. Бот задеплоен и **запущен один раз** (не две копии с одним токеном → ошибка 409 в логах).
2. В переменных окружения заданы `BOT_TOKEN`, `ADMIN_CHAT_ID`, `FIREBASE_DB_URL`, `FIREBASE_SERVICE_ACCOUNT_JSON`.
3. На бесплатном хостинге (Render Free) сервис не «уснул».
4. В логах после старта есть строки: `✅ Firebase connected`, `🧹 Webhook cleared`, `🤖 Bot started`.

## 🛠 Итерация 4 — исправления по логам

### Бот падал при обрыве сети (ECONNRESET)
В логах был `[polling_error] EFATAL: read ECONNRESET` — соединение с Telegram оборвалось и polling больше не восстанавливался (бот «жив», но не отвечает). Теперь бот **сам перезапускает polling** при сетевых сбоях (`ECONNRESET`, `ETIMEDOUT`, `EFATAL` и т.п.), а на конфликт 409 — наоборот, не перезапускается, чтобы не воевать со второй копией. Добавлены `keepAlive` и перехват `uncaughtException`.
> На бесплатных хостингах сеть «моргает» регулярно — теперь это не убивает бота. Если используете Render Free, учтите, что сервис ещё и засыпает без активности: для бота лучше платный план или Railway.

### Service Worker падал на chrome-extension запросах
Ошибка из консоли `Failed to execute 'put' on 'Cache': Request scheme 'chrome-extension' is unsupported` — SW пытался кэшировать запросы расширений браузера. Теперь SW обрабатывает только `http(s)`-запросы, а `cache.put` защищён. (кэш `v6`)

### Предупреждения Firmware про индексы
`FIREBASE WARNING: Using an unspecified index … add ".indexOn": "createdAt"` — это предупреждение о производительности (без индекса БД скачивает все записи и фильтрует на клиенте). Добавлен файл **`database.rules.json`** с нужными индексами.

**Как применить правила:**
1. Firebase Console → ваш проект → **Realtime Database** → вкладка **Rules**.
2. Вставьте содержимое `database.rules.json` (можно без строки `_comment`).
3. Нажмите **Publish**.

⚠️ Файл сохраняет ОТКРЫТЫЙ доступ (`.read/.write: true`), потому что клиентский портал читает БД без входа (по токену из ссылки). Не закрывайте доступ на `auth != null` — иначе портал перестанет работать без отдельной доработки авторизации.

### Мелочь
Добавлен современный `<meta name="mobile-web-app-capable">` рядом с устаревшим apple-тегом (убирает warning в консоли).

## 🆕 Итерация 5 — авто-уведомления и управление из бота

### Авто-уведомления теперь работают надёжно
Раньше уведомление о новой заявке могло не прийти из-за привязки к времени старта бота. Переписано: при старте бот один раз помечает все существующие заявки/отчёты как «уже уведомлённые» (чтобы не спамить бэклогом), а дальше реагирует на КАЖДУЮ новую запись. Теперь:
- **Новая заявка с сайта** → админу сразу приходит уведомление.
- **Новый отчёт** → админу приходит уведомление с кнопкой «✈ Отправить клиенту».

### Управление статусами прямо из чата (inline-кнопки)
- Под уведомлением о заявке — кнопки **«🔧 В работу»** и **«✅ Выполнено»**. Нажатие меняет статус заявки в базе (сайт видит сразу, realtime).
- Под уведомлением об отчёте — кнопка **«✈ Отправить клиенту»**: бот отправляет отчёт клиенту в Telegram на его языке и помечает отчёт отправленным.

### Поле «Telegram Chat ID» в карточке клиента
Теперь клиента можно подключить к боту вручную, даже если у него нет @username:
1. Клиент пишет боту `/start` → бот отвечает «ваш chat ID: …».
2. Впишите этот ID в карточке клиента (Изменить → Telegram Chat ID).
3. Клиент подключён — будет получать напоминания и отчёты.

> Если у клиента есть @username и он указан в карточке — подключение по-прежнему происходит автоматически при `/start`.

## 🆕 Итерация 6 — клиент узнаёт о смене статуса заявки

Теперь когда статус заявки меняется (кнопкой в боте «🔧 В работу»/«✅ Выполнено» ИЛИ на сайте) — клиенту автоматически приходит уведомление в Telegram на его языке:
- «🔧 Ваша заявка взята в работу…»
- «✅ Ваша заявка выполнена…»

Работает только если клиент подключён к боту (🟢). Уведомление отправляется один раз на каждое изменение статуса (защита от дублей). Реализовано через единый `child_changed` listener, поэтому срабатывает одинаково и из бота, и с сайта.

## 🆕 Итерация 7 — аналитика, экспорт, автосчета, безопасность

### Дашборд-аналитика
На Дашборде добавлен блок «Аналитика»: доход по тарифам (Basic/Standard/Premium), состояние объектов (норма/внимание/проблема), выполнение визитов за текущий месяц.

### Поиск и экспорт
- Поиск по клиентам (имя, страна, телефон, Telegram) и по отчётам (по адресу). Поиск по объектам уже был.
- Экспорт в CSV: кнопка «⬇ CSV» на вкладках Клиенты и Финансы (открывается в Excel, разделитель `;`, с BOM для кириллицы).

### Авто-счета и напоминания об оплате (в боте)
Бот раз в час проверяет:
- **1-го числа месяца** — автоматически создаёт счета по всем объектам (без дублей) и присылает админу сводку.
- **Просроченные счета** — pending-счёт из прошлого месяца помечается overdue, и клиенту один раз уходит напоминание об оплате в Telegram на его языке.
> Формат ID счёта одинаков у сайта и бота, поэтому ручная и авто-генерация не конфликтуют.

### Безопасность портала (требует действия в Firebase!)
Клиентский портал теперь входит **анонимно** (`Auth.ensureAnonAuth`), что позволяет закрыть базу правилом `auth != null`. Файл `database.rules.json` обновлён до защищённой версии.

**⚠️ Чтобы это заработало, выполните ДВА шага в Firebase Console (иначе портал перестанет читать данные):**
1. **Authentication → Sign-in method → Anonymous → Enable.**
2. **Realtime Database → Rules** → вставьте содержимое `database.rules.json` (без строки `_comment_ru`) → **Publish.**

После этого база закрыта от анонимного доступа из интернета: читать/писать могут только админ (email), портал (анонимно) и бот (service account).

## 🛠 Итерация 8 — исправления

1. **Бот: переключение статуса заявки.** После «🔧 В работу» теперь появляется кнопка «✅ Выполнено» (раньше кнопка пропадала и завершить из бота было нельзя). После «Выполнено» кнопки убираются.
2. **Отправка отчёта клиенту («✈ Отправить в Telegram»)** переделана на надёжный путь через бота: сайт ставит флаг `sendToClient`, бот доставляет отчёт клиенту (текст на его языке + фото по публичным URL). Прямая отправка из браузера оставлена как fallback. Если у клиента нет Telegram или токена — теперь показывается понятная причина, а не тишина.
3. **Фото во втором отчёте.** Исправлен сброс формы: при открытии нового отчёта плейсхолдер загрузки фото возвращается, и фото снова прикрепляются (раньше после первого отчёта поле фото оставалось «скрытым»).
4. **Доход клиента €0** — это не баг: доход считается из тарифов привязанных объектов. У клиентов без объектов он и есть €0. Появится после привязки объекта.

> Примечание про фото в Telegram: на бесплатном плане Firebase фото хранятся как base64 в базе (не публичный URL), поэтому бот отправляет клиенту текст отчёта, а фото — только если они доступны по http-ссылке. Полная отправка фото потребует Firebase Storage (платный план) или другого файлового хранилища.

## 🆕 Итерация 10 — уведомления о статусе, /help, дашборд-оплаты

1. **Уведомление админу при смене статуса с сайта.** Когда статус заявки меняется в приложении — бот присылает админу короткое «ℹ️ Статус заявки изменён». Если статус меняли кнопкой в самом боте — повторного уведомления нет (админ и так это видел). Клиенту уведомление приходит в обоих случаях.
2. **Бот: команда `/help`** — показывает список команд (для админа — админские, для клиента — подсказку по заявке).

4. **Дашборд: блок «Оплаты»** — если есть просроченные/ожидающие счета, на дашборде появляется карточка с суммами и переходом в Финансы (просроченные подсвечены красным).

## 🛠 Итерация 12 — доработка светлой темы

Светлая тема переработана: раньше фон body и часть градиентов были «зашиты» тёмными — поэтому в светлой теме карточки становились белыми на тёмном фоне (выглядело сломанно). Теперь:
- Фон страницы использует переменную темы (светлый фон в светлой теме).
- Hero и карточка «Объектов» получили корректные светлые градиенты (синий на белом).
- Палитра светлой темы сделана тёплой и мягкой (не режущий чисто-белый), золотой акцент затемнён для читаемости на светлом.

> Клиентский портал пока остаётся в тёмной теме (у клиентов нет переключателя). При желании можно добавить и туда.

### Как проверить выставление счетов
- **Бот:** команда `/testbilling` — создаёт счета за текущий месяц и обрабатывает просрочки сразу.
- **Сайт:** Финансы → «＋ Счета за месяц».
- Результат смотреть в разделе Финансы (статусы, кнопки оплаты, PDF). Автоматически счета создаются 1-го числа каждого месяца.
